# Phase 1: Plugin Scaffold and Tooling - Research

**Researched:** 2026-04-21
**Domain:** Claude Code plugin architecture, SKILL.md specification, CLI tooling (CJS)
**Confidence:** HIGH

## Summary

Phase 1 creates the complete plugin directory structure, stub SKILL.md files for all `/ttm-*` commands, the `bin/ttm-tools.cjs` CLI utility, dual-runtime support file templates, and the `.marketing/` directory template. This is a greenfield scaffold phase with zero external dependencies -- every output is a file created from scratch using only Node.js built-ins and Markdown.

The SKILL.md frontmatter specification has been verified against the official Claude Code documentation (code.claude.com/docs/en/skills, fetched 2026-04-21). The plugin.json manifest has been verified against the plugins reference (code.claude.com/docs/en/plugins-reference). GSD's `gsd-tools.cjs` (1,158 lines) has been directly inspected for subcommand routing patterns, JSON output conventions, and utility function implementations.

**Primary recommendation:** Mirror GSD's architecture exactly for `ttm-tools.cjs` (single entry point, `output()` helper for JSON/raw dual-mode, `error()` for stderr+exit, `lib/` subdirectory for growing modules). For SKILL.md stubs, use minimal frontmatter (name + description + disable-model-invocation: true) with placeholder bodies that reference future phase implementation.

## Architectural Responsibility Map

| Capability | Primary Tier | Secondary Tier | Rationale |
|------------|-------------|----------------|-----------|
| Plugin manifest + discovery | Plugin root (.claude-plugin/) | -- | Claude Code auto-discovers plugin.json at this path |
| Slash command registration | Skills layer (skills/ttm-*/SKILL.md) | -- | Each directory becomes a `/ttm-*` command via SKILL.md frontmatter |
| Workflow orchestration | Workflows layer (workflows/**/*.md) | -- | Commands route to workflows; workflows contain process logic |
| Deterministic operations | CLI layer (bin/ttm-tools.cjs) | -- | Slug generation, timestamps, state parsing must be deterministic, not AI-generated |
| User project state | Filesystem (.marketing/) | -- | All mutable state in user project directory, not plugin directory |
| Reference knowledge | Templates/references layer | -- | Static domain knowledge bundled with plugin, loaded on demand |
| Dual-runtime instructions | Generated files (CLAUDE.md/AGENTS.md) | -- | Generated into user project by /ttm-init, not bundled |

### Pattern 2: ttm-tools.cjs Subcommand Router

**What:** Single CJS entry point with switch/case routing to handler functions.
**When to use:** All deterministic operations.
**Example:**

```javascript
// Source: Verified pattern from ~/.claude/get-shit-done/bin/gsd-tools.cjs
#!/usr/bin/env node

const fs = require('fs');
const path = require('path');
const { execSync } = require('child_process');

// ─── Output helpers ──────────────────────────────────────────────────────
function output(result, raw, rawValue) {
  if (raw && rawValue !== undefined) {
    process.stdout.write(String(rawValue));
  } else {
    process.stdout.write(JSON.stringify(result, null, 2));
  }
  process.stdout.write('\n');
}

function error(message) {
  process.stderr.write('Error: ' + message + '\n');
  process.exit(1);
}

// ─── Commands ────────────────────────────────────────────────────────────
function cmdSlug(text, raw) {
  if (!text) error('text required for slug generation');
  const slug = text
    .toLowerCase()
    .replace(/[^a-z0-9]+/g, '-')
    .replace(/^-+|-+$/g, '')
    .substring(0, 60);
  output({ slug }, raw, slug);
}

function cmdTimestamp(format, raw) {
  const now = new Date();
  let result;
  switch (format) {
    case 'date': result = now.toISOString().split('T')[0]; break;
    case 'filename': result = now.toISOString().replace(/:/g, '-').replace(/\..+/, ''); break;
    default: result = now.toISOString(); break;
  }
  output({ timestamp: result }, raw, result);
}

// ─── Router ──────────────────────────────────────────────────────────────
const args = process.argv.slice(2);
const raw = args.includes('--raw');
const command = args[0];

switch (command) {
  case 'slug':
    cmdSlug(args.slice(1).filter(a => a !== '--raw').join(' '), raw);
    break;
  case 'timestamp':
    cmdTimestamp(args[1] || 'full', raw);
    break;
  case 'init':
    // Validate .marketing/ exists, return JSON state
    break;
  case 'state':
    // Subcommand routing: read, update
    break;
  case 'commit':
    // Git commit helper
    break;
  case 'health':
    // Validate .marketing/ directory integrity
    break;
  default:
    error(`Unknown command: ${command}. Available: slug, timestamp, init, state, commit, health`);
}
```

### Anti-Patterns to Avoid

- **Putting skills inside `.claude-plugin/`:** Only `plugin.json` goes there. Skills, agents, bin/ go at plugin root. [VERIFIED: official docs explicitly warn about this] 
- **Monolithic SKILL.md:** Keep each under 100 lines. Route to workflows for logic. Claude loads SKILL.md into context on invocation -- large files waste tokens.
- **AI-generated slugs/timestamps:** These must be deterministic. Always use `bin/ttm-tools.cjs` via Bash tool.
- **npm dependencies in ttm-tools.cjs:** Must use Node.js built-ins only (fs, path, crypto, child_process). Zero deps.
- **Bundling CLAUDE.md/AGENTS.md with plugin:** These are generated per-project by `/ttm-init`, not shipped with the plugin.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Slug generation | Custom regex per workflow | `node ttm-tools.cjs slug "text"` | Must be deterministic and consistent across all commands |
| Timestamps | `new Date()` in AI context | `node ttm-tools.cjs timestamp` | AI-generated dates can be hallucinated |
| State parsing | Manual YAML frontmatter parsing | Centralized state read/write in ttm-tools.cjs | Atomic updates prevent race conditions |
| Git commits | Inline git commands | `node ttm-tools.cjs commit "msg" --files f1 f2` | Consistent commit formatting, file staging |
| Plugin manifest | Hand-written JSON | Copy verified schema from this research | Easy to miss required fields or structure |

## Common Pitfalls

### Pitfall 1: Skills Directory Placement
**What goes wrong:** Skills placed inside `.claude-plugin/` are not discovered by Claude Code.
**Why it happens:** Intuition says plugin components go inside the plugin metadata directory.
**How to avoid:** Only `plugin.json` goes in `.claude-plugin/`. Everything else at plugin root.
**Warning signs:** `/ttm-*` commands don't appear in Claude Code's skill list.

### Pitfall 2: Frontmatter `name` vs Directory Name
**What goes wrong:** Skill name in frontmatter doesn't match directory name, causing confusion about which name is used.
**Why it happens:** Claude Code uses the `name` field from frontmatter if present, otherwise falls back to directory name.
**How to avoid:** Always set `name` in frontmatter to match directory name exactly (e.g., directory `ttm-init`, frontmatter `name: ttm-init`).
**Warning signs:** Command shows up under unexpected name.

### Pitfall 3: Description Truncation
**What goes wrong:** Skill description is truncated in Claude's context, losing critical trigger phrases.
**Why it happens:** Combined description + when_to_use is capped at 1,536 characters per skill. Total budget is 1% of context window.
**How to avoid:** Front-load the key use case in the first sentence of description. Keep under 200 characters for stubs.
**Warning signs:** Claude doesn't auto-invoke skills when expected (not relevant for Phase 1 since all stubs use `disable-model-invocation: true`).

### Pitfall 4: ttm-tools.cjs File Size Growth
**What goes wrong:** Single file grows past 500 lines, violating FOUND-03.
**Why it happens:** Adding subcommands directly to main file (GSD's gsd-tools.cjs is 1,158 lines in the main file but uses `lib/` modules for heavy lifting).
**How to avoid:** Start with `lib/` subdirectory from day one. Main `ttm-tools.cjs` contains only the router and output helpers. All command implementations in `lib/*.cjs`.
**Warning signs:** `ttm-tools.cjs` approaching 300 lines.

### Pitfall 5: Tier 1 Summaries Too Verbose
**What goes wrong:** "Compact" summaries exceed 200 words, eating into the 2,000-token budget.
**Why it happens:** Trying to include all positioning detail in the summary.
**How to avoid:** Template enforces strict structure: category (1 line), audience (1 line), differentiator (1 line), proof points (3-5 bullets), must-not-say (key terms only).
**Warning signs:** Tier 1 summaries exceed 150 words each. With 9 reference files, that's 1,350 words = ~1,800 tokens, leaving no margin.

### Pitfall 6: REQUIREMENTS.md vs CONTEXT.md Conflict on .marketing/ Structure
**What goes wrong:** REQUIREMENTS.md FOUND-07 says "CAMPAIGNS/, PLAYBOOKS/" but CONTEXT.md D-20 only shows CAMPAIGNS/.
**Why it happens:** Requirements were written before discuss phase refined the structure. Playbooks are bundled with the plugin (in `playbooks/`), not in the user's `.marketing/`.
**How to avoid:** Follow CONTEXT.md D-20 -- `.marketing/` has CAMPAIGNS/ only. Playbooks live in the plugin's `playbooks/` directory.
**Warning signs:** Creating a `PLAYBOOKS/` directory inside `.marketing/`.

### GSD output() Pattern for JSON/Raw Dual-Mode

```javascript
// Source: ~/.claude/get-shit-done/bin/lib/core.cjs (directly inspected)
function output(result, raw, rawValue) {
  if (raw && rawValue !== undefined) {
    // Raw mode: plain text for Bash tool consumption
    process.stdout.write(String(rawValue) + '\n');
  } else {
    // Default: JSON for structured consumption by AI
    const json = JSON.stringify(result, null, 2);
    if (json.length > 50000) {
      // Large payloads: write to tmpfile, output @file: prefix
      const tmpPath = path.join(GSD_TEMP_DIR, `gsd-${Date.now()}.json`);
      fs.writeFileSync(tmpPath, json, 'utf-8');
      process.stdout.write('@file:' + tmpPath + '\n');
    } else {
      process.stdout.write(json + '\n');
    }
  }
}

function error(message) {
  process.stderr.write('Error: ' + message + '\n');
  process.exit(1);
}
```

## Open Questions

1. **@ file reference scope in SKILL.md**
   - What we know: SKILL.md supports supporting files within its own directory (e.g., `reference.md` next to SKILL.md). The docs say "Reference these files from your SKILL.md so Claude knows what each file contains and when to load them."
   - What's unclear: Whether `@workflows/lifecycle/brief.md` syntax works for files outside the skill's own directory (e.g., at the plugin root level). GSD uses this pattern but it may rely on Claude reading the path as an instruction rather than a built-in resolution mechanism.
   - Recommendation: Use explicit `Read` tool instructions in SKILL.md body (e.g., "Read and follow the workflow at `${CLAUDE_SKILL_DIR}/../../workflows/lifecycle/brief.md`") rather than relying on `@` syntax. Test with a simple skill first. The `${CLAUDE_SKILL_DIR}` variable is verified to resolve to the skill's SKILL.md directory.

2. **Skill description budget with 25+ commands**
   - What we know: Total skill description budget is 1% of context window (~2,000 tokens for 200K context). Each description capped at 1,536 chars.
   - What's unclear: With 25+ stub skills all having `disable-model-invocation: true`, are their descriptions excluded from the budget? Official docs say "disable-model-invocation: true" means "Description not in context."
   - Recommendation: This is actually fine. Per the verified frontmatter reference table, skills with `disable-model-invocation: true` have "Description not in context." Only the few advisory skills (health, positioning-check) without this flag will consume budget. [VERIFIED: official docs]
