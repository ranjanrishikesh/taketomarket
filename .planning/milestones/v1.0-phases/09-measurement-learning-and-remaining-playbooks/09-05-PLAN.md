**<process> Steps:**

## Step 1: Load Context
Display: `takeToMarket > LOADING CAMPAIGN CONTEXT FOR LEARNING`
- Read Tier 1 summary blocks from all 9 .marketing/ reference files
- Read Tier 2 (full content) for: .marketing/LEARNINGS.md, .marketing/METRICS.md, .marketing/BRAND.md, .marketing/ICP.md, .marketing/CHANNELS.md, .marketing/COMPETITORS.md, .marketing/POSITIONING.md
- Read full campaign directory artifacts per the scan order from learnings-extraction.md:
  - `.marketing/CAMPAIGNS/${SLUG}/STATE.md` (frontmatter for gate results, run counts)
  - `.marketing/CAMPAIGNS/${SLUG}/MEASUREMENT.md` (measurement report with outcome data)
  - `.marketing/CAMPAIGNS/${SLUG}/BRIEF.md` (original strategy and targets)
  - `.marketing/CAMPAIGNS/${SLUG}/VERIFY-REPORT-*.md` (gate details if they exist)
  - `.marketing/CAMPAIGNS/${SLUG}/FIX-BRIEF-*.md` (fix details if they exist)
  - `.marketing/CAMPAIGNS/${SLUG}/REVIEW-FEEDBACK-*.md` (reviewer comments if they exist)

## Step 2: Validate Campaign State
```bash
node "${CLAUDE_PLUGIN_ROOT}/bin/ttm-tools.cjs" campaign state ${SLUG} --raw
```
- Verify campaign is in `measured` or `learned` phase (re-learning allowed)
- If not measured: display error "Campaign must be measured before learning. Current phase: ${PHASE}. Run /ttm-measure first."

## Step 3: Extract Outcome Deltas (per LRNG-02)
Compare measurement results against brief targets:
- For each metric in MEASUREMENT.md, calculate delta vs target from BRIEF.md
- Classify each delta: overperformed (>10% above target), met (within +/-10%), underperformed (>10% below target)
- List all deltas with classification

Display:
```
takeToMarket > OUTCOME DELTAS

| Metric | Target | Actual | Delta | Classification |
|--------|--------|--------|-------|----------------|
| [outcome metric] | [target] | [actual] | [+/-X%] | [over/met/under] |
| [output metric 1] | ... | ... | ... | ... |
```

## Step 4: Classify Lessons per Root-Cause Taxonomy (per LRNG-01, LIFE-17)
For each underperforming or failing element, classify using the 7-category root-cause taxonomy from learnings-extraction.md:
- positioning-drift
- weak-hook
- wrong-channel
- bad-timing
- unverifiable-claim
- broken-funnel
- creative-fatigue

For each overperforming element, classify as "success" with a description of what pattern drove the win.

Scan campaign artifacts in the order specified by learnings-extraction.md:
1. STATE.md gate fields for pass/fail patterns
2. VERIFY-REPORT files for specific gate failure details
3. FIX-BRIEF files for root-cause analysis already performed
4. REVIEW-FEEDBACK files for human reviewer signals
5. BRIEF.md for strategic context

Generate 1 lesson per notable delta (both successes and failures). Each lesson is one specific, actionable sentence -- not a generic observation.

## Step 5: Propose Reference File Edits (per D-07, D-08, D-09, LIFE-16)
For each lesson that implies a systemic change, propose an edit using the narrative + apply approach per D-07:

For each proposed edit:

1. **Present the narrative:**
   "Based on campaign [slug] results, [specific observation]. [Reasoning from data]. Therefore, [specific edit proposal]."

2. **Show the specific edit:**
   ```
   File: .marketing/[TARGET_FILE].md
   Section: [section name]
   Change: [specific addition, modification, or removal]
   ```

3. **Ask for approval (AskUserQuestion or text-mode numbered list per D-08):**
   ```
   Apply this to [FILE]?
   1. Apply -- Update [FILE] with this change
   2. Skip -- Keep [FILE] unchanged, log lesson only
   3. Modify -- Edit the proposed change before applying
   ```

4. **POSITIONING.md special handling per D-09:**
   If the proposed edit targets POSITIONING.md, do NOT offer direct apply. Instead display:
   ```
   This edit targets POSITIONING.md, which is locked during campaigns.
   This change requires /ttm-positioning-shift.
   1. Launch positioning shift -- Start /ttm-positioning-shift with this proposal
   2. Skip -- Log the lesson but do not change POSITIONING.md
   ```

5. **If user selects "Apply":** Read the target file, find the specified section, apply the edit, write the updated file. Track as edits_applied.

6. **If user selects "Modify":** Ask user for the modified version, then apply.

7. **If user selects "Skip":** Log the lesson but set Action Taken to "none -- skipped by user."

Valid edit targets per D-09: BRAND.md, ICP.md, CHANNELS.md, POSITIONING.md (via shift only), METRICS.md, COMPETITORS.md.

## Step 6: Append Lessons to LEARNINGS.md (per LRNG-01, LIFE-17)
1. Read `.marketing/LEARNINGS.md`
2. Find the marker line: `<!-- LESSONS BELOW THIS LINE -->`
3. Insert new lesson rows immediately after the marker, one row per lesson from Step 4:
   ```
   | ${DATE} | ${SLUG} | ${CATEGORY} | ${LESSON_TEXT} | ${ACTION_TAKEN} |
   ```
4. Update the Summary section counters at the top of the file:
   - Total lessons: [increment by number of new lessons]
   - Last lesson date: [today's date]
   - Top pattern: [update if the most frequent category changed]
5. Write updated LEARNINGS.md

## Step 7: Pattern Extraction (per LRNG-03)
Count the number of distinct campaign slugs in the LEARNINGS.md Lessons Log table.

If fewer than 3 campaigns have entries:
```
takeToMarket > PATTERN EXTRACTION

Pattern extraction requires 3+ campaigns with lessons. Current: ${N}.
Skipping pattern extraction -- will run automatically after ${3 - N} more campaigns.
```

If 3 or more campaigns have entries:
1. Scan all lesson rows for frequency patterns
2. Update the Pattern Extraction sections:
   - **Winning Hooks:** Hooks/openings that appeared in "success" lessons across 2+ campaigns
   - **Winning Angles:** Positioning angles or themes with consistent overperformance
   - **Winning Formats:** Content formats with best outcome metric results
   - **Anti-Patterns:** Approaches that appeared in failure lessons across 2+ campaigns
3. Replace the "[Populated after 3+ campaigns...]" placeholder text with actual patterns
4. Write updated LEARNINGS.md
