`<process>`:

**Step 1: Load Context** -- Banner: `takeToMarket > LOADING CONTEXT`
- Extract SLUG from $ARGUMENTS: `SLUG=$(echo "$ARGUMENTS" | sed 's/--text//g' | xargs)`
- If SLUG is empty, error: "Usage: /ttm-resume [campaign-slug]. Provide a campaign slug." Exit.

**Step 2: Load Campaign State** -- Banner: `takeToMarket > LOADING CAMPAIGN STATE`
- Run: `node "${CLAUDE_PLUGIN_ROOT}/bin/ttm-tools.cjs" campaign state "${SLUG}" --raw`
- If `exists: false`: tell user campaign not found, suggest `/ttm-new-campaign`. Exit.
- Parse the full JSON output. Extract: `phase`, `name`, `last_updated`, all `phase.*` timestamps, `fix.run_count`, `review.overall_result`, `verify.overall_result`, gate results.
- Also read the FULL STATE.md file (not just CLI output) to get body content per Pitfall 5: `Read .marketing/CAMPAIGNS/${SLUG}/STATE.md` -- the body contains "Phase:" and "Next step:" lines plus any additional notes.

**Step 3: Build Recovery Context** -- Banner: `takeToMarket > RECOVERY CONTEXT`
- Determine last completed phase by finding the latest non-null `phase.*` timestamp. Use this ordering: created, researched, briefed, produced, verified, reviewed, fixed, shipped, measured, learned.
- For each completed phase, note the timestamp.
- Calculate time since last activity from `last_updated`.

**Step 4: Detect Interrupted Loops** (per D-05)
- Check if campaign is mid-fix-loop: `fix.run_count` exists and is > 0, AND `review.overall_result` equals `'revise'`.
  - If yes: flag as interrupted fix loop. Suggest `/ttm-fix ${SLUG}` to continue (not restart).
- Check if campaign is mid-verify: `verify.run_count` exists and is > 0, AND `verify.overall_result` is NOT `'pass'`.
  - If yes: suggest re-running `/ttm-verify ${SLUG}`.

**Step 5: Determine Next Command**
- Use the phase-to-command mapping (same as /ttm-next):
  - `created` -> `/ttm-research ${SLUG}`
  - `researched` -> `/ttm-brief ${SLUG}`
  - `briefed` -> `/ttm-produce ${SLUG}`
  - `produced` -> `/ttm-verify ${SLUG}`
  - `verified` -> `/ttm-review ${SLUG}`
  - `reviewed` -> if `review.overall_result = 'revise'` -> `/ttm-fix ${SLUG}`, if `'approved'` -> `/ttm-ship ${SLUG}`
  - `fixed` -> `/ttm-review ${SLUG}`
  - `shipped` -> `/ttm-measure ${SLUG}` (note: Phase 9)
  - `measured` -> `/ttm-learn ${SLUG}` (note: Phase 9)
  - `learned` -> `/ttm-archive ${SLUG}`
  - `archived` -> "Campaign already archived. No further action needed."
  - `cancelled` -> "Campaign cancelled. No further action. Create a new campaign with `/ttm-new-campaign`."

**Step 6: Display Recovery Summary** -- Banner: `takeToMarket > RESUME SUMMARY`
- Display formatted summary per D-04:
```
## Campaign Resume: ${SLUG}

**Campaign:** ${name}
**Current Phase:** ${phase}
**Last Activity:** ${relative_time} (${last_updated})

### Completed Phases
| Phase | Completed At |
|-------|-------------|
| Created | 2026-04-20 |
| Researched | 2026-04-21 |
| ... | ... |

### Pending Work
${next_step_description}

### Suggested Next Command
> `/ttm-fix spring-launch`
> Reason: Fix loop in progress (attempt 2 of 3). Review flagged positioning drift.

${if_interrupted_loop: "**Note:** This campaign has an interrupted fix loop. Running /ttm-fix will continue from attempt ${fix.run_count}, not restart."}
```

`<process>`:

**Step 1: Load Context** -- Banner: `takeToMarket > LOADING CONTEXT`
- Extract SLUG from $ARGUMENTS: `SLUG=$(echo "$ARGUMENTS" | sed 's/--text//g' | xargs)`
- If SLUG is empty, error: "Usage: /ttm-archive [campaign-slug]. Provide a campaign slug." Exit.

**Step 2: Validate Campaign** -- Banner: `takeToMarket > VALIDATING CAMPAIGN`
- Run: `node "${CLAUDE_PLUGIN_ROOT}/bin/ttm-tools.cjs" campaign state "${SLUG}" --raw`
- If `exists: false`: tell user campaign not found. Exit.
- Check `phase` field:
  - If `'archived'`: tell user campaign is already archived. Exit.
  - If `'cancelled'`: tell user cancelled campaigns cannot be archived per D-08. They stay in CAMPAIGNS/ as cautionary records. Exit.
  - If NOT `'shipped'`: tell user only shipped campaigns can be archived. Show current phase and suggest completing remaining phases. Exit.

**Step 3: Extract Learnings** -- Banner: `takeToMarket > EXTRACTING LEARNINGS`
- Following the learnings-extraction.md reference guide, scan campaign artifacts:
  - Read `.marketing/CAMPAIGNS/${SLUG}/STATE.md` (full file -- frontmatter and body)
  - Read `.marketing/CAMPAIGNS/${SLUG}/MANIFEST.json` if it exists (per-asset gate results)
  - Read `.marketing/CAMPAIGNS/${SLUG}/BRIEF.md` if it exists (original strategy)
  - Read any `VERIFY-REPORT-*.md` files in the campaign directory (gate details)
  - Read any `FIX-BRIEF-*.md` files (fix attempts and outcomes)
  - Read any `REVIEW-FEEDBACK-*.md` files (human review feedback)

- Extract structured learnings in three categories per D-09:
  1. **What worked:** First-attempt gate passes, high review scores, assets shipped without fix loops
  2. **What didn't work:** Gate failures categorized by root-cause taxonomy (positioning-drift, weak-hook, wrong-channel, bad-timing, unverifiable-claim, broken-funnel, creative-fatigue)
  3. **Campaign-level decisions:** Positioning anchor, channel mix rationale, hook strategy, any deviations accepted during verify

- Format each learning as a LEARNINGS.md table row:
  `| ${today} | ${SLUG} | ${category} | ${lesson} | ${action_taken} |`

${extracted_rows}

Campaign directory moved to `.marketing/CAMPAIGNS/ARCHIVE/${SLUG}/`

| Boundary | Description |
|----------|-------------|
| User input -> workflow SLUG extraction | Slug passed to CLI for archive operation |
| Campaign artifacts -> learnings extraction | AI reads potentially corrupted campaign files |
| Workflow -> LEARNINGS.md write | Append operation must not corrupt existing content |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-09 | Tampering | SLUG in archive workflow | mitigate | Pass through CLI which sanitizes slug and checks path escape. Workflow does not construct filesystem paths directly. |
| T-07-10 | Tampering | LEARNINGS.md marker-based append | mitigate | Check for exactly 1 marker occurrence before appending. If 0 or >1, fall back to end-of-table append. Follows drift-log.cjs pattern. |
| T-07-11 | Denial of Service | Large campaign directory copy | accept | `fs.cpSync` is bounded by campaign directory size. Individual campaigns are small (Markdown files). |
| T-07-12 | Repudiation | Archive without learnings | mitigate | Archive workflow always extracts learnings before moving. CLI archive requires shipped phase. User must confirm. |
</threat_model>
