<objective>
Build the /ttm-review workflow that presents verified assets to the user with a structured review checklist, collects per-asset outcomes (Approve/Revise/Reject), persists structured revision feedback for assets marked Revise, and auto-triggers /ttm-fix for revised assets.

Purpose: This is the human-in-the-loop quality gate -- automated verification (Phase 4) catches measurable issues, but human review catches strategic misalignment, tone problems, and creative quality that gates cannot. No asset ships without human review.

Output: `workflows/lifecycle/review.md` -- complete review workflow.
</objective>

**Structure the workflow with these sections:**

```
<purpose>
Review workflow for /ttm-review. Presents verified assets with structured
review checklist for human evaluation (LIFE-10). Collects per-asset outcomes
(Approve/Revise/Reject per D-03) with hero-first ordering (D-02). Structured
revision feedback on Revise (D-12). Auto-triggers /ttm-fix for revised assets (D-15).
</purpose>

<required_reading>
@${CLAUDE_PLUGIN_ROOT}/references/context-loading.md
@${CLAUDE_PLUGIN_ROOT}/references/review-checklist.md
</required_reading>
```

**Step 1: Text-Mode Detection** -- Same pattern as verify.md:
- Check `$ARGUMENTS` for `--text` flag
- If `AskUserQuestion` unavailable, set `TEXT_MODE=true`
- Text-mode replaces AskUserQuestion with numbered list prompts

**Step 2: Load Context** -- Extract SLUG from $ARGUMENTS:
- Load Tier 1 summaries from all 9 reference files (per context-loading.md)
- Tier 2: None for review (per context-loading.md matrix -- review is human-driven)
- Load campaign-specific files (always full): `STATE.md`, `BRIEF.md`
- Load `MANIFEST.json` -- error if not found ("Run /ttm-produce first")
- Load `VERIFICATION.md` -- error if not found ("Run /ttm-verify first")

**Step 3: Validate Campaign State** -- Check via ttm-tools.cjs:
```bash
node "${CLAUDE_PLUGIN_ROOT}/bin/ttm-tools.cjs" campaign state "${SLUG}" --raw
```
- If phase is NOT `"verified"`: Warn "Campaign is in phase '${PHASE}'. Expected 'verified' before review. Proceed?" Wait for confirmation.
- Determine review run number from `review.run_count` (null/0 -> 1, else +1).

**Step 4: Load Assets and Verification Results**
- Parse MANIFEST.json for hero + derivatives (same pattern as verify.md Step 3)
- Parse VERIFICATION.md for the gate summary table -- extract per-asset PASS/WARN/FAIL counts and overall result
- For each asset, read the first ~500 characters from the file on disk (D-04)
- Build a combined view: asset name, gate summary, content preview, file path

**Step 5: Review Hero Asset (D-02)**
Display hero asset in full detail:
```
takeToMarket > REVIEW: Hero Asset

## [HERO_ASSET_NAME]
File: .marketing/CAMPAIGNS/${SLUG}/${HERO_FILE}

### Gate Summary
| Gate | Result |
|------|--------|
| Positioning Drift (GATE-01) | [PASS|WARN|FAIL] |
| ... (all 10 gates) |

### Content Preview
[First ~500 characters of the asset]
[Link to full file]

### Review Checklist
```
Present the 4 mandatory questions from review-checklist.md using AskUserQuestion (or text-mode). Each question must be answered before proceeding.

After all 4 questions answered, present freeform notes prompt (optional).

Then present outcome selection:
```
Select outcome for [HERO_ASSET_NAME]:
1. Approve -- asset is ready for shipping
2. Revise -- asset needs changes (you will provide structured feedback next)
3. Reject -- asset is fundamentally wrong (final, no fix loop)
```

If outcome is **Approve**: Set hero `review_status` = `"approved"` in memory.

If outcome is **Revise**: Collect structured revision feedback per D-12:
1. "Which review questions did this asset fail?" (multi-select: 1-4)
2. "Severity?" (1=Minor, 2=Major, 3=Critical)
3. "What specifically needs to change?" (freeform text)
Store the structured feedback. Set hero `review_status` = `"needs-fix"` in memory.
Write the structured feedback to `.marketing/CAMPAIGNS/${SLUG}/REVIEW-FEEDBACK-[ASSET_NAME].md`:
```markdown
# Review Feedback: [ASSET_NAME]

**Campaign:** [SLUG]
**Date:** [ISO_TIMESTAMP]
**Reviewer outcome:** Revise
**Severity:** [Minor|Major|Critical]

## Failed Checklist Items
- [x] Positioning reinforcement (if selected)
- [x] Outcome realism (if selected)
- [x] Claim substantiation (if selected)
- [x] Competitor differentiation (if selected)

## Specific Feedback
[User's freeform feedback text]

## Gate Failures (from VERIFICATION.md)
[Extract relevant WARN/FAIL findings for this asset]
```

If outcome is **Reject**: Prompt for rejection reason. Set hero `review_status` = `"rejected"` in memory. Log the rejection reason.

**Step 6: Batch Review Derivatives (D-02, D-13)**
If there are derivative assets, present each in batch format:
```
takeToMarket > REVIEW: Derivative Assets

Reviewing ${N} derivatives. Hero asset for reference: ${HERO_FILE}
```
For each derivative:
- Show abbreviated gate summary (PASS count / WARN count / FAIL count -- not full table)
- Show first ~300 chars + file link
- Same 4 review questions (can be answered more briefly)
- Same 3 outcome options (Approve/Revise/Reject)
- If Revise: collect same structured feedback
- If Reject: collect rejection reason

**Step 7: Update MANIFEST.json with Review Status**
Read MANIFEST.json, add `review_status` field to hero and each derivative:
- `"approved"` for Approve outcomes
- `"needs-fix"` for Revise outcomes
- `"rejected"` for Reject outcomes

Also add `review_feedback_file` for Revise outcomes pointing to the feedback file.

Write the updated MANIFEST.json back to disk.

**Step 8: Update Campaign State**
```bash
TIMESTAMP=$(node "${CLAUDE_PLUGIN_ROOT}/bin/ttm-tools.cjs" timestamp --raw)
node "${CLAUDE_PLUGIN_ROOT}/bin/ttm-tools.cjs" campaign update "${SLUG}" review.run_count ${RUN_NUMBER}
node "${CLAUDE_PLUGIN_ROOT}/bin/ttm-tools.cjs" campaign update "${SLUG}" review.last_run "$TIMESTAMP"
node "${CLAUDE_PLUGIN_ROOT}/bin/ttm-tools.cjs" campaign update "${SLUG}" review.overall_result "[approved|mixed|needs-fix]"
node "${CLAUDE_PLUGIN_ROOT}/bin/ttm-tools.cjs" campaign update "${SLUG}" phase reviewed
node "${CLAUDE_PLUGIN_ROOT}/bin/ttm-tools.cjs" campaign update "${SLUG}" phase.reviewed "$TIMESTAMP"
```

Overall result logic:
- `approved` -- all assets approved
- `needs-fix` -- all non-rejected assets need fix
- `mixed` -- some approved, some need fix, some rejected

**Step 9: Display Completion and Auto-Trigger Fix (D-15)**
```
takeToMarket > REVIEW COMPLETE

Run: ${RUN_NUMBER} | Date: ${ISO_DATE}
Assets reviewed: ${TOTAL}
  Approved: ${APPROVE_COUNT}
  Needs fix: ${REVISE_COUNT}
  Rejected: ${REJECT_COUNT}

[If any rejected:]
Rejected assets logged. These will not enter the fix loop.

[If any need fix:]
Launching /ttm-fix ${SLUG} for ${REVISE_COUNT} asset(s)...
```

If any assets were marked Revise, inform the user that /ttm-fix should be run next:
"Run `/ttm-fix ${SLUG}` to fix ${REVISE_COUNT} asset(s) with review feedback."

IMPORTANT per D-15: The auto-trigger means telling the user to run /ttm-fix next (since review.md is NOT forked and cannot directly invoke /ttm-fix as a separate command). The workflow completes with a clear "Next step" instruction.
