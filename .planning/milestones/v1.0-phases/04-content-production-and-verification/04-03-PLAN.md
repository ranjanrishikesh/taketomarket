**Per-gate specifications:**

**GATE-01: Positioning Drift (Tier 1)** — Checks against: POSITIONING.md
3 checks: (1) Primary differentiator alignment — asset reinforces or naturally extends the differentiator vs introduces new claims. (2) Proof point sourcing — all claims backed by POSITIONING.md proof point library. (3) Must-not-say compliance — no banned terms present. PASS/WARN/FAIL per check.

**GATE-02: Claim Accuracy (Tier 1)** — Checks against: BRAND.md proof points section
3 checks: (1) Every factual/numeric claim has a matching proof point in BRAND.md. (2) Proof point sources are cited (not just asserted). (3) No expired or deprecated proof points used. FAIL = any unverifiable claim.

**GATE-03: Voice Drift (Tier 2)** — Checks against: BRAND.md voice archetype + banned words
3 checks: (1) Tone matches the voice archetype defined in BRAND.md (e.g., "authoritative but approachable"). (2) No banned words from BRAND.md banned words list. (3) Language register is consistent (no sudden shifts from formal to casual). FAIL = banned word found.

**GATE-04: Outcome Alignment (Tier 1)** — Checks against: CAMPAIGNS/<slug>/BRIEF.md outcome metric
2 checks: (1) Asset is designed to drive the outcome metric defined in the brief (not just output). (2) CTA/action aligns with the outcome, not a tangential goal. FAIL = asset has no connection to outcome metric.

**GATE-05: Funnel Integrity (Tier 2)** — Checks against: BRIEF.md funnel/CTA section
3 checks: (1) CTA is present and specific (not vague "learn more"). (2) CTA destination is defined (URL or next step). (3) Conversion path from CTA to outcome is logical (no dead ends). N/A for assets without CTAs (informational content).

**GATE-06: UTM Hygiene (Tier 2)** — Checks against: CHANNELS.md UTM schema
3 checks: (1) All trackable links have UTM parameters. (2) UTM source/medium match CHANNELS.md naming conventions. (3) UTM campaign tag matches the campaign slug. N/A for assets without links.

**GATE-07: Compliance (Tier 2)** — Checks against: industry-standard requirements
3 checks: (1) Required disclaimers present (financial advice, health claims, affiliate disclosure). (2) No PII exposed (email addresses, phone numbers in body copy). (3) Unsubscribe/opt-out mechanism referenced where applicable (email, SMS). N/A for assets in channels without regulatory requirements.

**GATE-08: Competitor Collision (Tier 2)** — Checks against: COMPETITORS.md
3 checks: (1) Asset does not use competitor brand names without substantiated comparative claims. (2) Asset framing does not accidentally echo a competitor's positioning or tagline. (3) Asset differentiates rather than inadvertently validates competitor approach.

**GATE-09: ICP Fit (Tier 2)** — Checks against: ICP.md
3 checks: (1) Hook addresses a pain point from ICP.md pains list. (2) Language uses customer vocabulary from ICP.md language library, not internal jargon. (3) Content addresses the ICP segment, not the anti-ICP.

**GATE-10: Format Correctness (Tier 2)** — Checks against: channel-specific rules (from playbook if loaded, otherwise general platform guidelines)
3 checks: (1) Character/word count within platform limits (e.g., tweet under 280 chars, LinkedIn under 3000). (2) Required elements present (subject line for email, title for blog, hook for social). (3) Structure follows channel conventions (e.g., email has preview text, blog has H1).

```markdown
# Gate Evaluation Reference

## Usage

Referenced by `workflows/lifecycle/verify.md` via @-syntax.
Applied to each asset during verification. Each gate is evaluated independently
with its specific reference data loaded.

**Key rules:**
- Evaluate gates one at a time, not all bundled in a single prompt (per Pitfall 4)
- Load the gate's reference file(s) for each evaluation
- Use the structured output format below for every finding
- Soft fail on all gates per D-04 — report results, never block

---

## Structured Output Format

Every gate evaluation MUST produce output in this exact format:

### Gate Result
- **Gate:** [Gate Name] (GATE-XX)
- **Tier:** [1|2]
- **Result:** [PASS|WARN|FAIL]
- **Summary:** [One-sentence summary of the finding]

### Findings
[For each check within the gate:]

**Check: [check name]**
- **Result:** PASS | WARN | FAIL
- **Evidence:** "[exact quote from the asset that triggered this finding]" (line/section reference)
- **Reference:** "[quote from the reference file being compared against]"
- **Recommendation:** [what to change if WARN or FAIL; "None" if PASS]

If a check is N/A (e.g., UTM hygiene for an asset without links), record:
- **Result:** N/A
- **Evidence:** "Not applicable — [reason]"

---

## Per-Gate Evaluation Instructions

[For each of the 10 gates, provide:]

### Evaluating GATE-XX: [Gate Name]

**Load:** [reference file path(s)]
**Asset content:** [full asset text]

**Prompt the evaluation:**
[Specific questions to ask about the asset, referencing the evaluation criteria from base-gates.md]

**Tier:** [1|2]
**On failure:** [Tier 1: prompt user for Correct/Accept+log/Escalate | Tier 2: report as advisory]
```
