<objective>
Build the /ttm-ship workflow that generates a dynamic launch checklist per campaign based on channel mix and asset types, auto-checks verifiable items (UTM format, tracking presence, gate status), collects human confirmation for unverifiable items, and tracks per-asset ship status.

Purpose: This is the final quality gate before content goes live. Ship ensures nothing launches without tracking installed, UTMs confirmed, funnels tested, and all verification/review requirements met. The dynamic checklist eliminates irrelevant items (email campaigns don't see SEO checks) while ensuring nothing channel-specific is missed.

Output: `workflows/lifecycle/ship.md` -- complete ship workflow.
</objective>

**Step 1: Text-Mode Detection** -- Same pattern as verify.md.

**Step 2: Load Context** -- Extract SLUG from $ARGUMENTS:
- Load Tier 1 summaries from all 9 reference files
- Load Tier 2: `.marketing/CHANNELS.md` (per context-loading.md -- needed for UTM schema and channel details)
- Load campaign-specific files (full): `STATE.md`, `BRIEF.md`
- Load `MANIFEST.json` -- error if not found
- Load `VERIFICATION.md` -- error if not found

**Step 3: Validate Campaign State**
```bash
node "${CLAUDE_PLUGIN_ROOT}/bin/ttm-tools.cjs" campaign state "${SLUG}" --raw
```
- If campaign does not exist: error, suggest /ttm-new-campaign.
- Check that review has been completed: review.overall_result must not be null.
- If review.overall_result is null: "Campaign has not been reviewed. Run /ttm-review first." Exit.
- If fix is needed and fix.overall_result is null: Warn "Some assets need fixing. Run /ttm-fix first, or proceed to ship only approved assets?" Wait for confirmation.

**Step 4: Identify Ship-Ready Assets**
Parse MANIFEST.json. Categorize each asset by `review_status`:
- `"approved"` or `"ship-ready"` -> eligible for shipping
- `"needs-fix"` or `"needs-human-fix"` -> NOT eligible, display warning
- `"rejected"` -> excluded, already final
- `"shipped"` -> already shipped, skip

If no assets are eligible for shipping:
"No ship-ready assets found. All assets are either awaiting fix, rejected, or already shipped." Exit.

Display ship-ready assets:
```
takeToMarket > SHIP READINESS

Ship-ready assets: ${COUNT}
[List each ship-ready asset name + type + channel]

[If any not ready:]
Not ready (excluded from this ship run):
- [ASSET_NAME]: [STATUS] -- [reason]
```

**Step 5: Generate Dynamic Checklist (D-09)**
Read `${CLAUDE_PLUGIN_ROOT}/references/ship-checklist-items.md`.
From BRIEF.md, identify the campaign's channel mix (asset types and channels).

Build the checklist by combining:
1. **Universal section** (always included -- all campaigns)
2. **Channel-specific sections** -- for each unique asset type in the ship-ready list:
   - Match asset type to the corresponding section in ship-checklist-items.md
   - Blog/SEO if any `blog-post` type assets
   - Email if any `email` type assets
   - LinkedIn if any `linkedin-post` type assets
   - Social if any `social-post` or `twitter-post` type assets
   - Landing Page if any `landing-page` type assets
   - Video/YouTube if any `video` or `youtube-video` type assets
   - Paid Ads if any `paid-ad` type assets
   - Default section for any unmatched types

**Step 6: Run AI Auto-Checks (D-10)**
For each [AI]-tagged checklist item, attempt automated verification:

**Universal AI checks:**
- UTM parameter validity: Scan each ship-ready asset file for URLs containing `utm_` parameters. Validate format: `utm_source`, `utm_medium`, `utm_campaign` all present. Check against CHANNELS.md UTM schema if available. Result: PASS (valid UTMs found), WARN (no trackable links found), FAIL (malformed UTMs).
- Draft marker detection: Scan each asset for strings: `TODO`, `PLACEHOLDER`, `TBD`, `DRAFT`, `[INSERT`, `XXX`. Result: PASS (none found), FAIL (markers found with locations).
- Verification status: Read VERIFICATION.md frontmatter `overall_result`. Result: PASS if `pass` or `accepted`, FAIL otherwise.
- Review status: Check MANIFEST.json -- all ship-ready assets have `review_status` of `approved` or `ship-ready`. Result: PASS or FAIL.

**Channel-specific AI checks** (examples):
- Blog/SEO: Check first H1 exists, meta description exists (check frontmatter or first paragraph), check for orphaned links
- Email: Check subject line length (frontmatter `subject` field), check for "unsubscribe" string
- LinkedIn: Check post length (< 3000 chars), check opening line does not start with "I"
- Social: Check character count against platform limits
- Landing Page: Check for CTA presence (look for `[CTA]`, button-like markup, or action verbs)

For each AI check, record: item_name, result (PASS/WARN/FAIL), detail (what was found).

**Step 7: Present Checklist with Results**
Display the complete checklist organized by section. For each item:
- [AI] items: Show result (PASS/WARN/FAIL) with detail
- [HUMAN] items: Show as unchecked checkbox for user to confirm

```
takeToMarket > LAUNCH CHECKLIST: ${SLUG}

## Universal Checks
- [PASS] UTM parameters valid on all trackable links
- [PASS] All asset files finalized (no draft markers)
- [PASS] VERIFICATION.md shows pass/accepted
- [PASS] All ship-ready assets have approved review status
- [ ] Tracking/analytics configured for outcome metric -- CONFIRM? (yes/no)
- [ ] Monitoring and alerts set up -- CONFIRM? (yes/no)
- [ ] Team notified of launch timeline -- CONFIRM? (yes/no)

## Blog / SEO Checks
- [PASS] Meta title present
- [WARN] Meta description: first paragraph is 180 chars (recommend under 160)
- [ ] Schema markup deployed -- CONFIRM? (yes/no)
...
```

For each [HUMAN] item, use AskUserQuestion (or text-mode) to collect confirmation:
```
Confirm: [CHECKLIST_ITEM]
1. Yes -- confirmed
2. No -- not ready yet
3. N/A -- not applicable to this campaign
```

Track which items are confirmed, not ready, or N/A.

**Step 8: Determine Ship Decision**
After all items processed:
- If all [AI] checks PASS and all [HUMAN] items confirmed (yes or N/A): "All clear to ship!"
- If any [AI] FAIL or [HUMAN] "not ready": Display blockers:
  ```
  Blockers preventing ship:
  - [ITEM_NAME]: [REASON]

  Resolve these items and re-run /ttm-ship ${SLUG}
  ```
  Ask user: "Ship anyway with documented exceptions, or resolve blockers first?"
  ```
  1. Ship anyway -- proceed with documented exceptions
  2. Resolve first -- exit and fix blockers
  ```
  If "Ship anyway": proceed. Log exceptions.
  If "Resolve first": exit workflow.

**Step 9: Update Per-Asset Ship Status (D-11)**
For each ship-ready asset being shipped:
Read MANIFEST.json, update the asset entry:
- Add `ship_status: "shipped"`
- Add `shipped_at: "[ISO_TIMESTAMP]"`

Assets NOT being shipped (not ready, user deferred) retain their current status.

Write updated MANIFEST.json.

**Step 10: Update Campaign State**
```bash
TIMESTAMP=$(node "${CLAUDE_PLUGIN_ROOT}/bin/ttm-tools.cjs" timestamp --raw)
node "${CLAUDE_PLUGIN_ROOT}/bin/ttm-tools.cjs" campaign update "${SLUG}" ship.status "[shipped|partial]"
node "${CLAUDE_PLUGIN_ROOT}/bin/ttm-tools.cjs" campaign update "${SLUG}" ship.shipped_at "$TIMESTAMP"
node "${CLAUDE_PLUGIN_ROOT}/bin/ttm-tools.cjs" campaign update "${SLUG}" ship.checklist_result "[all-clear|exceptions]"
```

ship.status logic:
- `shipped` -- all ship-ready assets shipped
- `partial` -- some assets shipped, others deferred or blocked

Only update phase to "shipped" if at least one asset shipped:
```bash
node "${CLAUDE_PLUGIN_ROOT}/bin/ttm-tools.cjs" campaign update "${SLUG}" phase shipped
node "${CLAUDE_PLUGIN_ROOT}/bin/ttm-tools.cjs" campaign update "${SLUG}" phase.shipped "$TIMESTAMP"
```

**Step 11: Display Completion**
```
takeToMarket > SHIP COMPLETE

Date: ${ISO_DATE}
Assets shipped: ${SHIPPED_COUNT}/${TOTAL_SHIP_READY}

Shipped:
- [ASSET_NAME] (shipped at [TIMESTAMP])
- ...

[If partial:]
Deferred (not shipped yet):
- [ASSET_NAME]: [reason]

[If any exceptions:]
Shipped with exceptions:
- [EXCEPTION_ITEM]: [justification]

Checklist result: [all-clear | exceptions]

Next: Run /ttm-measure ${SLUG} when measurement window begins
      (or /ttm-ship ${SLUG} again to ship deferred assets)
```
