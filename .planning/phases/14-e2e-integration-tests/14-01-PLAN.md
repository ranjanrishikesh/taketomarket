    **Local helper (inside the test file per D-16, NOT in helpers.cjs):**
    ```javascript
    /**
     * Run install.js as a child process and capture stdout/stderr/exit.
     * Returns { stdout, stderr, status } even on non-zero exit (caught from error.status).
     * @param {string[]} args - CLI args passed after install.js
     * @param {object} env - Full env object (caller must include process.env spread + HOME override)
     */
    function runInstall(args, env) {
      try {
        const stdout = execFileSync('node', [INSTALL_JS, ...args], {
          env,
          cwd: PROJECT_ROOT,
          timeout: 30000,
          encoding: 'utf-8',
          stdio: ['ignore', 'pipe', 'pipe'],
        });
        return { stdout, stderr: '', status: 0 };
      } catch (err) {
        return {
          stdout: err.stdout ? err.stdout.toString() : '',
          stderr: err.stderr ? err.stderr.toString() : '',
          status: typeof err.status === 'number' ? err.status : 1,
        };
      }
    }

    /**
     * Build env object with isolated HOME pointing at the given temp dir.
     * Spreads process.env so PATH and Node infra remain intact.
     */
    function envWithHome(tmpDir) {
      return { ...process.env, HOME: tmpDir };
    }
    ```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| test runner → child node process | Test code spawns `node install.js` with controlled args + env; arg array form prevents shell-injection class threats |
| install.js → real filesystem | Without HOME override, install.js would write to the developer's real `~/.claude/` — that would be a real-environment-pollution incident, not a security incident, but it is the primary safety concern of this phase |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-01 | Tampering | runInstall execFileSync invocation | mitigate | Use `execFileSync('node', [INSTALL_JS, ...args], { env, cwd, ... })` argument-array form; never construct shell strings or use `execSync(\`node ... ${userInput}\`)`. Test args are hardcoded literals so injection is theoretical, but the array form is the correct pattern even for hardcoded data. |
| T-14-02 | Information Disclosure | HOME env override | mitigate | Every `runInstall` call MUST pass `envWithHome(tmp.dir)` — never bare `process.env`. A code review check (acceptance criterion 3 grep gate) confirms `HOME:` appears in the file. If a future test ever omits the override, install.js would write to the developer's real home dir; the per-describe lifecycle + helper-only invocation pattern minimizes this risk. |
| T-14-03 | Denial of Service | child process hang | mitigate | `timeout: 30000` (30s) on every execFileSync call. install.js is sync I/O bounded by the source tree size (~200 files); 30s is generous. Acceptance criterion 10 grep gates this. |
| T-14-04 | Tampering | temp dir cleanup | accept | Each describe block calls `tmp.cleanup()` in `after`. If a test crashes between `before` and `after`, the temp dir leaks under `os.tmpdir()` — accepted risk since OS-managed temp dirs get reclaimed and contain no sensitive data (only fixture install output). |
</threat_model>
