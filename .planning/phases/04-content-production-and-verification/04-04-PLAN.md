<objective>
Create the /ttm-verify workflow that evaluates every produced asset against 10 quality gates with structured deviation handling.

Purpose: This is the quality gate wall — the core invariant enforcer of takeToMarket. Every asset must pass through 10 gates before progressing to review. The verify workflow runs in a context completely isolated from produce (via context:fork) to prevent self-evaluation bias (per LIFE-09). It loads assets from disk, evaluates each against all 10 gates, presents findings as a summary table with drill-down, and handles deviations with 3 options per D-09. This plan depends on Plan 01 (state fields) and Plan 03 (gate definitions).

Output: Updated SKILL.md with context:fork, verify.md workflow with gate evaluation loop, deviation handling, and verification report generation.

**Steps inside `<process>` (9 steps):**

**Text-Mode Detection** — copy the exact pattern from brief.md lines 18-43. Verify DOES use AskUserQuestion for deviation handling prompts, so text-mode fallback is important.

**Step 1: Load Context** — follow brief.md Step 1 pattern:
- Display: `takeToMarket > LOADING CONTEXT`
- Extract slug: `SLUG=$(echo "$ARGUMENTS" | sed 's/--text//g' | xargs)`
- Load Tier 1 summaries from all 9 reference files
- Load Tier 2 full content for: POSITIONING.md, BRAND.md, ICP.md (needed for gate evaluation)
- Read campaign files: CAMPAIGNS/${SLUG}/STATE.md, CAMPAIGNS/${SLUG}/BRIEF.md
- Read MANIFEST.json: `.marketing/CAMPAIGNS/${SLUG}/MANIFEST.json`
- If MANIFEST.json not found, error: "No production manifest found. Run /ttm-produce first."

**Step 2: Validate Campaign State** — follow brief.md campaign validation pattern:
- Run `node "${CLAUDE_PLUGIN_ROOT}/bin/ttm-tools.cjs" campaign state "${SLUG}" --raw`
- Check `exists: false` -> error
- Check `phase` field -> warn if not "produced" (expected predecessor for verify)
- Determine verify run number: read `verify.run_count` from state, increment by 1 (or set to 1 if null)

**Step 3: Load Assets from Disk** — per D-11, LIFE-09:
- Display: `takeToMarket > LOADING ASSETS`
- Parse MANIFEST.json to get list of asset files
- Read each asset file from `.marketing/CAMPAIGNS/${SLUG}/[asset.file]`
- If any asset file is missing, report warning and skip that asset
- Store asset contents for gate evaluation
- CRITICAL: Assets are loaded from DISK only. Never from produce context. This is the self-evaluation bias prevention (per D-10, LIFE-09).

**Step 4: Evaluate Gates Per Asset** — The core verification loop:
- Display: `takeToMarket > EVALUATING QUALITY GATES`
- For each asset:
  - For each of the 10 gates (follow order from base-gates.md):
    - Follow the evaluation instructions from gate-evaluation.md
    - Load the gate's reference file(s) as specified in gate-evaluation.md
    - Evaluate the asset content against the gate criteria
    - Record the structured output: gate name, ID, tier, result (PASS/WARN/FAIL), findings[]
    - Each finding has: check name, result, evidence (exact quote), reference, recommendation
  - After all 10 gates evaluated for an asset, aggregate: count PASSes, WARNs, FAILs
- IMPORTANT per RESEARCH.md Pitfall 4: Evaluate each gate SEPARATELY, not bundled. Load gate-specific reference data for each evaluation.

**Step 5: Build Summary Table** — per D-05:
- Display: `takeToMarket > BUILDING VERIFICATION REPORT`
- Construct the summary table from templates/verification-report.md format:
  - Header row: #, Gate, Tier, then one column per asset
  - One row per gate (10 rows) showing PASS/WARN/FAIL/N-A per asset
  - Result summary: count of Tier 1 FAILs, total WARNs
- Display the summary table to the user

**Step 6: Handle Tier 1 Deviations** — per D-04, D-09, GATE-12:
- For each Tier 1 gate (GATE-01, GATE-02, GATE-04) that returned WARN or FAIL on any asset:
  - Display the detailed finding (evidence + reference + recommendation)
  - Present 3 options using AskUserQuestion (or text-mode numbered list):
    ```
    Gate [name] returned [WARN|FAIL] for [asset]:
    [finding detail with evidence and reference]

    Choose an action:
    1. Correct — Record this for /ttm-fix to address
    2. Accept+log — Document exception and proceed
    3. Escalate — Launch /ttm-positioning-shift
    ```
  - Process user's choice:
    - **Correct (1):** Record finding as needing fix. Continue verification.
    - **Accept+log (2):** Prompt for justification via AskUserQuestion: "Why are you accepting this deviation?" Then write deviation record to DEVIATIONS.md (append-only per D-07) with all required fields: gate, asset, tier, result, finding, action, justification, timestamp, verify_run. Also record in STATE.md via campaign update.
    - **Escalate (3):** Display "Launching /ttm-positioning-shift. Verification paused. After positioning is resolved, re-run /ttm-verify." Stop verification (per D-08).
- For the DEVIATIONS.md append: If file doesn't exist yet, copy from template. Then append the new entry following the template's entry format. Use `node "${CLAUDE_PLUGIN_ROOT}/bin/ttm-tools.cjs" timestamp --raw` for consistent timestamps.

**Step 7: Display Tier 2 Findings** — per D-04:
- For each Tier 2 gate (GATE-03, GATE-05 through GATE-10) that returned WARN or FAIL:
  - Display as advisory: "[Advisory] Gate [name]: [finding summary] | Recommendation: [recommendation]"
- No action required for Tier 2, but user can optionally request action

**Step 8: Write Verification Report** — persist results:
- Write VERIFICATION.md to `.marketing/CAMPAIGNS/${SLUG}/VERIFICATION.md`
- Use the templates/verification-report.md format
- Include: run number, date, summary table, all findings detail, actions taken
- If VERIFICATION.md already exists from a prior run, overwrite (report is current state, not append-only — unlike DEVIATIONS.md)

**Step 9: Update Campaign State and Summary** — follow brief.md state update pattern:
- Update all 10 gate result fields:
  ```bash
  node "${CLAUDE_PLUGIN_ROOT}/bin/ttm-tools.cjs" campaign update "${SLUG}" gate.positioning_drift [pass|warn|fail]
  node "${CLAUDE_PLUGIN_ROOT}/bin/ttm-tools.cjs" campaign update "${SLUG}" gate.claim_accuracy [pass|warn|fail]
  # ... all 10 gates
  ```
- Update verification metadata:
  ```bash
  node "${CLAUDE_PLUGIN_ROOT}/bin/ttm-tools.cjs" campaign update "${SLUG}" verify.run_count [N]
  node "${CLAUDE_PLUGIN_ROOT}/bin/ttm-tools.cjs" campaign update "${SLUG}" verify.last_run "$TIMESTAMP"
  node "${CLAUDE_PLUGIN_ROOT}/bin/ttm-tools.cjs" campaign update "${SLUG}" verify.overall_result [pass|warn|fail]
  ```
- Update campaign phase (only if no Escalate was triggered):
  ```bash
  node "${CLAUDE_PLUGIN_ROOT}/bin/ttm-tools.cjs" campaign update "${SLUG}" phase verified
  node "${CLAUDE_PLUGIN_ROOT}/bin/ttm-tools.cjs" campaign update "${SLUG}" phase.verified "$TIMESTAMP"
  ```
- Overall result logic: "pass" if all gates pass, "warn" if any warn but no fail, "fail" if any Tier 1 fail (even if Accept+logged)
- Display completion banner:
  ```
  takeToMarket > VERIFICATION COMPLETE

  Run: [N] | Result: [PASS/WARN/FAIL]
  Assets verified: [count]
  Tier 1 failures: [count] ([count] corrected, [count] accepted, [count] escalated)
  Tier 2 advisories: [count]

  Report: .marketing/CAMPAIGNS/${SLUG}/VERIFICATION.md
  Deviations: .marketing/CAMPAIGNS/${SLUG}/DEVIATIONS.md

  ```
