**Step 1: Pre-flight** -- Implement exactly as specified in PATTERNS.md:

```markdown
## Step 1: Pre-flight

```
takeToMarket > INITIALIZING
```

Run init status check:
```bash
node ${CLAUDE_PLUGIN_ROOT}/bin/ttm-tools.cjs init --raw
```

**If result is "initialized":**

Use AskUserQuestion:
- header: "Existing Setup"
- question: ".marketing/ already contains reference files. What would you like to do?"
- options:
  - label: "Start fresh"
    description: "Delete existing files and re-run the full interview"
  - label: "Update specific files"
    description: "Choose which reference files to regenerate"
  - label: "Cancel"
    description: "Exit without changes"

**If "Start fresh":** Delete .marketing/ contents (except CAMPAIGNS/) and continue.
**If "Update specific files":** Present file picker using AskUserQuestion with multiSelect: true listing all 9 reference files. Run only the interview sections that map to selected files (use the Question-to-Template Mapping from init-questions.md).
**If "Cancel":** Exit with message "No changes made."

**If result is "not initialized":**
```bash
mkdir -p .marketing/CAMPAIGNS
mkdir -p .marketing/PLAYBOOKS
```
Continue to Step 2.
```

**Steps 2-7: Interview Sections** -- For each section:

1. Display stage banner: `takeToMarket > SECTION N: [NAME]`
2. Ask the structured questions (AskUserQuestion format) from init-questions.md
3. Ask the freeform questions from init-questions.md
4. Run specificity validation per init-validation.md rules for that section
5. If validation fails: re-prompt using the Re-prompt Template from init-validation.md (max 2 retries, then accept with SPECIFICITY_WARNING)
6. Store collected answers for file generation in Step 9

Important for each section's prompt text:
- Reference which template file the section maps to (e.g., "This section collects data for POSITIONING.md")
- Include the freeform questions verbatim from init-questions.md (do not just say "ask the questions from init-questions.md")
- Include the validation criteria inline (do not just say "run validation from init-validation.md") -- the executor needs the actual rules visible

**Step 8: Confirmation Gate** -- Before generating files:

```markdown
## Step 8: Confirmation Gate

Display a summary of what was collected:
- Product category and positioning summary
- Voice archetype
- Primary ICP segment
- Active channels
- Top competitors
- Primary outcome metric

Use AskUserQuestion:
- header: "Ready?"
- question: "Here's what I'll use to generate your reference files. Ready to proceed?"
- options:
  - label: "Generate files"
    description: "Create all .marketing/ reference files"
  - label: "Revise a section"
    description: "Go back and update specific answers"

If "Revise a section": Use AskUserQuestion with section list, re-run that section, return to confirmation.
```

**Step 9: Generate Reference Files** -- For each of the 9 files:

```markdown
## Step 9: Generate Reference Files

For each reference file, follow this pattern:

1. Read the template: Read file `${CLAUDE_PLUGIN_ROOT}/templates/reference-files/{filename}.md`
2. Generate the file content by filling ALL `[GENERATED BY /ttm-init]` placeholders with interview data
3. **Critical structural requirements:**
   - Preserve `<!-- _SUMMARY: ... -->` and `<!-- END_SUMMARY -->` comment markers exactly as they appear in the template
   - Summary section must be under 200 words
   - Preserve all table structures from the template
   - For dates, use: `node ${CLAUDE_PLUGIN_ROOT}/bin/ttm-tools.cjs timestamp date --raw`
4. Write to `.marketing/{FILENAME}.md`

**Special handling:**
- **STATE.md:** Copy template and update frontmatter `last_updated` field via timestamp command. Do NOT fill interview data into STATE.md -- it is initialized with default values.
- **LEARNINGS.md:** Copy template verbatim. Set Summary fields to "Total lessons: 0", "Last lesson date: none", "Top pattern: none". Do NOT fill interview data -- learnings are populated during campaign Learn phases.

Generate files in this order:
1. POSITIONING.md (from Section 1 data)
2. BRAND.md (from Section 2 data)
3. ICP.md (from Section 3 data)
4. CHANNELS.md (from Section 4 data)
5. COMPETITORS.md (from Section 5 data)
6. METRICS.md (from Section 6 data -- metrics portion)
7. CALENDAR.md (from Section 6 data -- calendar portion)
8. STATE.md (template copy with timestamp)
9. LEARNINGS.md (template copy with zeroed counters)
```

**Step 10: Generate Instruction Files:**

## Step 10: Generate Instruction Files

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User answers -> file generation | User-provided text is written to .marketing/ Markdown files. No code execution risk -- files are Markdown only. |
| Workflow -> filesystem | Workflow writes to .marketing/ and project root (CLAUDE.md, AGENTS.md). Path security handled by ttm-tools.cjs. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-03 | Tampering | File writes to .marketing/ | mitigate | Pre-flight check warns before overwriting existing files; user must confirm "Start fresh" |
| T-02-04 | Tampering | CLAUDE.md/AGENTS.md overwrite | mitigate | Workflow copies from static templates only; no user-supplied content injected into instruction files |
| T-02-05 | Denial of Service | Interview abandonment | accept | If user exits mid-interview, no files are written. State remains "not initialized". User re-runs /ttm-init to restart. |
</threat_model>
