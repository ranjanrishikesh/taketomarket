# Project Research Summary

**Project:** takeToMarket
**Domain:** Claude Code / Codex marketing operating system skill
**Researched:** 2026-04-21
**Confidence:** HIGH

## Executive Summary

takeToMarket is a marketing operating system built as a Claude Code plugin — not a SaaS, not a chatbot, but a structured skill installed into the user's AI environment that enforces a 9-phase campaign lifecycle. Research confirms the right way to build this is to follow the GSD (Get Shit Done) architectural pattern: thin SKILL.md entry points delegate to workflow files, all mutable state lives in a `.marketing/` directory in the user's project, and the entire system is zero-dependency Markdown with optional Node.js CJS tooling for deterministic operations. The Claude Code plugin format (`.claude-plugin/plugin.json`) is the primary distribution path, with npm and git clone as secondary options.

The recommended approach centers on three architectural invariants that separate takeToMarket from generic AI content tools: positioning-as-invariant (POSITIONING.md is read-only during campaigns, any change requires explicit `/ttm-positioning-shift` with human approval), outcome-over-output (every asset requires a measurable outcome metric defined at brief time, not retrospectively), and fresh-context production (assets are produced in isolated context windows to prevent quality degradation across a long session). These three invariants are not nice-to-have — they are the entire value proposition. A build that compromises any of them produces something indistinguishable from "just prompting Claude directly."

The primary risks are architectural, not feature-based. Context window bloat from loading all reference files into every command, self-evaluation bias when the same session produces and verifies content, and a monolithic SKILL.md that exceeds Claude's attention budget are all pitfalls that cause silent quality degradation — the skill appears to work but produces generic output. All three must be addressed in Phase 1 (Foundation), before any lifecycle commands are built. A modular file architecture with tiered context loading and separate Produce/Verify contexts is not premature optimization — it is the prerequisite for correctness.

## Key Findings

### Recommended Stack

The entire skill is Markdown-native with YAML frontmatter. There are no runtime dependencies — the AI runtime reads the files, so there is nothing to npm install. The only code is an optional `bin/ttm-tools.cjs` file using Node.js built-ins for deterministic operations (slug generation, state timestamps, atomic file writes). Claude Code plugin format handles distribution and namespacing. CLAUDE.md and AGENTS.md are generated by `/ttm-init` into the user's project for dual-runtime (Claude Code + Codex) compatibility. Skills must be placed at the plugin root level, not inside `.claude-plugin/` — a common and hard-to-debug mistake documented in official plugin-dev docs.

**Core technologies:**
- **SKILL.md (Agent Skills standard):** Skill entry point and slash command definition — universal standard across Claude Code, Codex, Cursor, and 8+ runtimes
- **Markdown + YAML frontmatter:** All skill content, templates, workflows, references — lingua franca of the Agent Skills ecosystem, the only viable format
- **Claude Code plugin format (.claude-plugin/):** Primary distribution — provides automatic namespacing (`/taketomarket:ttm-*`), marketplace discovery, and one-command updates
- **Node.js CJS (bin/ttm-tools.cjs):** Deterministic CLI utility — slug generation, state management, atomic writes; uses built-in modules only, no npm dependencies
- **CLAUDE.md + AGENTS.md:** Runtime instruction files generated by `/ttm-init` — dual-runtime compatibility; both live in the user's project root

### Expected Features

The feature research is definitive: the 9-phase campaign lifecycle IS the product. Shortcuts undermine it. Users will recognize the core table-stakes features immediately and will adopt takeToMarket specifically for the differentiators — no other tool enforces positioning as an architectural invariant or treats content failures like engineering bugs with root-cause analysis.

**Must have (table stakes — MVP):**
- Guided onboarding (`/ttm-init`) — generates all 6+ reference files via structured interview; first entry point, nothing else works without it
- 9-phase campaign lifecycle with slash commands per phase — what makes this a system, not a chat session
- Persistent campaign state (`CAMPAIGNS/<slug>/STATE.md`) — sessions end; work must survive
- Brand voice and positioning context loading into every phase — without this, output is generic
- Quality gate wall (10 base gates) with deviation reports — the verification backbone
- Fix-as-you-go with root cause analysis and 3-attempt cap — treats content failures like bugs
- State management (`/ttm-state`, `/ttm-resume`) — session recovery
- SEO, Email, LinkedIn playbooks — highest demand, most structured formats; sufficient for MVP validation

**Should have (differentiators — v1.x after validation):**
- Measure phase with manual analytics paste and structured analysis — closes the outcome loop
- Learn phase and LEARNINGS.md with root-cause taxonomy — the compounding value proposition
- Content repurposing pipeline (`/ttm-repurpose`) — long-form to cross-channel derivatives
- Meta-gates (portfolio balance, calendar collision) — strategic quality layer
- Remaining 7 discipline playbooks (AEO, YouTube, Social, Paid Ads, Affiliate, PR/Media, Events)
- Positioning shift workflow (`/ttm-positioning-shift`) — structured repositioning with migration plan
- Discover phase (`/ttm-research`) — research-informed briefs

**Defer (v2+):**
- MCP integrations (GA, GSC, HubSpot) — adds external dependencies; MVP works without them
- Team/concurrent editing — requires coordination infrastructure beyond file-based state
- Automated competitor monitoring — requires background execution, not a skill capability

**Anti-features (deliberately excluded):**
- Real-time analytics dashboard (SaaS feature, not skill feature)
- Direct publishing/scheduling integrations (fragile external dependencies)
- One-click campaign generation (defeats the spec-driven approach)
- Visual/image generation (out of scope; Claude Code is text-based)

### Architecture Approach

The architecture follows GSD's 7-layer model adapted for marketing: (1) Skill entry and commands, (2) Workflows (the orchestration brain), (3) Playbooks (discipline knowledge, NOT executable — they are context loaded by workflows), (4) Quality gates (base + discipline-specific + meta), (5) Templates (output scaffolds the AI fills in), (6) References (static domain knowledge: attribution models, funnel stages, root-cause taxonomy), and (7) User project state (`.marketing/` directory, mutable, git-tracked). The critical design principle is Separation of Skill from State: the installed skill package is read-only; all mutable campaign data lives in the user's project under `.marketing/`. POSITIONING.md is the architectural invariant — it loads into every phase context but only `/ttm-positioning-shift` can modify it, and only after human approval.

**Major components:**
1. **SKILL.md entry point** — lean routing file (under 300 lines); reads metadata, routes commands to workflows via `@workflows/` references; never contains inline workflow logic
2. **Slash commands (commands/ttm-*.md)** — one file per command, thin entry points only; each reads the relevant workflow file, never duplicates logic
3. **Workflows (workflows/)** — the actual multi-step orchestration logic, organized by category (lifecycle, setup, utility, reference-mgmt, discipline); reusable across commands (verify.md is called by both `/ttm-verify` and `/ttm-fix`)
4. **Playbooks (playbooks/)** — discipline-specific knowledge loaded as context during Produce and Verify; pure reference, not executable; each defines additive discipline-specific gates
5. **Gates (gates/)** — base-gates.md (10 universal), meta-gates.md (portfolio-level), discipline/ (per-playbook extensions); gates are tiered (Tier 1: blocking, Tier 2: advisory)
6. **Templates (templates/)** — output scaffolds for BRIEF.md, asset specs, verification reports, reference file onboarding output; the AI fills them, no template engine
7. **User project state (.marketing/)** — POSITIONING.md (invariant), BRAND.md, ICP.md, CHANNELS.md, COMPETITORS.md, METRICS.md, LEARNINGS.md, CALENDAR.md, STATE.md, plus CAMPAIGNS/ directory

### Critical Pitfalls

1. **Context window bloat from loading all reference files into every command** — Design a two-tier loading strategy: compact summaries (<200 words each) load universally; full documents load only in phases that need them. Budget explicitly: ~2,000 tokens of reference material maximum per command, leaving 95%+ of context for actual work. This must be decided in Phase 1 — retrofitting it is a rewrite.

2. **Self-evaluation bias: same session produces and verifies content** — Verify phase MUST run in a fresh context, never in the same session as Produce. Load only the output and evaluation criteria into the Verify context — never the production prompt or reasoning. Design gates as binary rubrics with specific checkable criteria, not subjective assessments. If all 10 gates pass on first attempt regularly, the gates are too lenient.

3. **Monolithic SKILL.md that exceeds Claude's attention budget** — No single file exceeds 500 lines. Each slash command is one minimal file. Adding a command requires creating 1 new file and 0 edits to existing files. Trigger descriptions must be short and keyword-dense. Modular architecture from day one — retrofitting modularity is a full rewrite.

4. **Positioning drift detection that is too fuzzy to be trusted** — POSITIONING.md must be structured as a checklist (primary differentiator phrase, category, target audience, must-include proof points, must-not-say collision terms), not prose. Drift detection checks specific fields, not the overall document. Implement a three-tier result (PASS / WARN / FAIL) — only FAIL blocks shipping. Log all WARNs to LEARNINGS.md for pattern review.

5. **Fix loop non-convergence: verify-fix cycles that never converge** — Tier gates so only Tier 1 (blocking) gates trigger the fix loop. When multiple gates fail, the fix brief must address all failures simultaneously, not one at a time. After 2 failed attempts, escalate to human review rather than running a 3rd automated fix. Track gate conflict patterns in LEARNINGS.md.

## Implications for Roadmap

Based on research, suggested phase structure:

### Phase 1: Foundation
**Rationale:** Architecture decisions made here cannot be changed without rewriting the skill. Context loading strategy, file modularity, state management design, and the positioning invariant schema must all be correct before any lifecycle code is written. Three pitfalls (context bloat, monolithic skill file, state corruption) are Phase 1 problems with HIGH recovery cost if deferred.
**Delivers:** Plugin structure, SKILL.md entry point, all templates and reference files, base gate definitions, state management design (two-level STATE.md with atomic writes), ttm-tools.cjs CLI utility, onboarding interview (`/ttm-init`) with specificity validation, CLAUDE.md + AGENTS.md dual-runtime generation
**Addresses:** All table-stakes infrastructure; zero feature work possible without this
**Avoids:** Context window bloat (tiered loading designed here), monolithic SKILL.md (modular architecture from day one), state corruption (atomic writes, append-only log, `/ttm-health` validator), low-quality onboarding output (specificity scoring and positioning stress test)

### Phase 2: Core Campaign Lifecycle
**Rationale:** The 9-phase lifecycle is the product. Once foundation is in place, build the lifecycle phases in dependency order (Brief to Produce to Verify to Review to Fix to Ship), enforcing fresh-context separation between Produce and Verify from the start. Measure and Learn phases come after because they require completed campaigns to have value.
**Delivers:** `/ttm-new-campaign`, `/ttm-brief` (with outcome metric enforcement), `/ttm-produce` (wave-parallel, fresh-context per asset), `/ttm-verify` (10 base gates, deviation reports, 3-option resolution), `/ttm-review` (human gate), `/ttm-fix` (root-cause brief, 3-attempt cap with Tier 1/2 gate tiering), `/ttm-ship` (publish-ready output with UTMs), `/ttm-state`, `/ttm-resume`. SEO, Email, LinkedIn playbooks built here to validate the playbook inheritance model with 2-3 disciplines before scaling.
**Addresses:** Campaign lifecycle (P1), positioning-as-invariant (P1), quality gate wall (P1), state persistence (P1), outcome-over-output enforcement (P1), fix-as-you-go (P1), deviation reports (P1)
**Avoids:** Self-evaluation bias (fresh-context architecture enforced), fix loop non-convergence (gate tiering and simultaneous multi-gate fix briefs), slash command sprawl (happy path uses 4-5 commands to complete first campaign), unmeasurable outcome metrics (metric taxonomy constrained to GA/GSC/email-platform measurables)
**Research flag:** The Produce workflow wave-parallel fresh-context pattern for multi-asset campaigns is the most architecturally complex component. Explicit design work needed before implementation to determine how to orchestrate context-forking, collect results, and merge back to campaign state.

### Phase 3: Playbook Expansion and Measurement Loop
**Rationale:** The core lifecycle validates the thesis. Phase 3 adds the compounding value: discipline-specific playbooks enrich production and verification, and the Measure+Learn loop creates the feedback system that makes the skill smarter over time. Playbook inheritance model validated with 2-3 playbooks in Phase 2; expand to all 10 here.
**Delivers:** Measure phase (`/ttm-measure` with 3 attribution models and manual paste workflow), Learn phase (`/ttm-learn` with root-cause taxonomy and reference file edit proposals), all 7 remaining discipline playbooks (AEO, YouTube, Social, Paid Ads, Affiliate, PR/Media, Events), discipline-specific gate extensions, playbook-specific measurable outcome metric taxonomies, LEARNINGS.md accumulation
**Addresses:** Measure phase (P2), Learn phase (P2), remaining playbooks (P2), compound learning system (core differentiator)
**Avoids:** Playbook explosion maintenance burden (inheritance model validated before scaling to 10 playbooks), unmeasurable outcome metrics per discipline (playbook-specific metric taxonomies mapped to real analytics tools)
**Research flag:** Each playbook must define outcome metrics measurable with the tools a solo marketer actually has (GA, GSC, email platform, LinkedIn Analytics). Mapping each discipline's aspirational outcomes to paste-able metrics requires per-channel domain research.

### Phase 4: Advanced Workflows and Meta-Features
**Rationale:** Features that require campaign history to have value. Meta-gates need multiple campaigns; repurposing needs verified source assets; positioning shift needs real drift patterns to respond to. Also the UX polish that reduces command sprawl for new users.
**Delivers:** Content repurposing pipeline (`/ttm-repurpose`), meta-gates (portfolio balance, calendar collision, theme consistency), positioning shift workflow (`/ttm-positioning-shift`), reference refresh commands (`/ttm-brand-refresh`, `/ttm-icp-refresh`, `/ttm-competitor-scan`), discover phase (`/ttm-research`), `/ttm-next` command (guides users to the right next command based on state), `/ttm-health` enhanced diagnostics, `/ttm-archive`
**Addresses:** Repurposing (P2), meta-gates (P2), positioning shift (P2), slash command sprawl/cognitive overload, discover phase (P2)
**Avoids:** Calendar collision bugs (CALENDAR.md collision detection tested with real campaign history), command confusion (` /ttm-next` makes lifecycle invisible for new users)

### Phase 5: Distribution and npm Polish
**Rationale:** Plugin format is the primary distribution path, but npm (`npx taketomarket`) reaches users who prefer it. Both must produce identical behavior. This is last because the skill must be stable before packaging for distribution.
**Delivers:** npm package with single `install.js` entry point, runtime detection (Claude Code vs Codex), file manifest tracking for updates, symlink creation to `~/.claude/skills/`, post-install validation script that runs `/ttm-health`, git clone documentation, dual-runtime parity test suite
**Addresses:** Plugin distribution (required for marketplace), npm distribution (secondary convenience path), Codex compatibility parity
**Avoids:** npm distribution misconfiguration (post-install validation), dual distribution parity failures (explicit parity testing)
**Research flag:** npm installer architecture for plugin-format skills installed via npm (vs. git clone with `claude --plugin-dir`) is less documented than git clone path. May need verification against plugin-dev skill documentation before implementation.

### Phase Ordering Rationale

- Foundation before lifecycle: three pitfalls have HIGH recovery cost if discovered late (monolithic skill, context loading, state management). Fixing them after lifecycle phases are built requires rewriting every command.
- Brief before Produce before Verify: strict data dependency — each phase's output is the next phase's input. Not a preference; it is the architecture.
- Produce and Verify built together in Phase 2: they must be built together to enforce fresh-context separation. Building them in separate phases risks the separation being treated as a later concern.
- Measure and Learn in Phase 3, not Phase 2: they require completed campaigns and real analytics data to validate. Building them before any campaigns complete produces untestable code.
- Playbooks expanded in Phase 3 after inheritance model validated: scaling to 10 before the model is proven creates the playbook explosion maintenance burden.
- Advanced workflows in Phase 4: meta-gates require multiple campaigns; repurposing requires verified source assets. Structurally dependent on Phase 2 and Phase 3 completing.
- Distribution last: the skill must be stable before packaging. A broken npm package is harder to recover from than a broken git clone.

### Research Flags

Phases likely needing deeper research during planning:
- **Phase 2 (Produce workflow):** Wave-parallel fresh-context production for multi-asset campaigns is the most architecturally complex component. How to orchestrate context-forking, collect results, and merge back to campaign state needs explicit design work before implementation.
- **Phase 3 (Discipline-specific measurable metrics):** Each playbook must define outcome metrics measurable with the tools a solo marketer actually has. Mapping each discipline's aspirational outcomes to paste-able metrics requires per-channel domain research.
- **Phase 5 (npm installer for plugin format):** Plugin-format skills installed via npm are less documented than git clone. Needs verification against plugin-dev skill documentation.

Phases with standard patterns (skip research-phase):
- **Phase 1 (Foundation structure):** Plugin format, SKILL.md frontmatter, template creation, bin/ tooling — all directly documented in Claude Code plugin-dev docs and GSD source. Research already complete in STACK.md and ARCHITECTURE.md.
- **Phase 4 (Utility commands):** `/ttm-state`, `/ttm-resume`, `/ttm-health`, `/ttm-archive` follow established GSD utility command patterns. Standard state-reading workflows with no novel complexity.

### Gaps to Address

- **Positioning drift detection calibration:** The PITFALLS.md prescribes a structured checklist approach but the exact threshold for PASS/WARN/FAIL needs real-world calibration against actual marketing content. Cannot be fully resolved before testing with real campaigns.
- **Fix loop convergence in practice:** Whether Tier 1 vs Tier 2 classification is correct for each of the 10 base gates needs validation against real content failures. Some gates may be mis-tiered on first attempt.
- **Context budget empirical testing:** The 2,000-token reference material budget is grounded in published research on LLM context degradation, but the optimal split for takeToMarket's specific reference files needs empirical testing during Phase 2.
- **Codex parity:** Testing that all 20+ slash commands work identically across Claude Code and Codex is deferred to Phase 5. Assume potential surprises on runtime-specific behavior.
